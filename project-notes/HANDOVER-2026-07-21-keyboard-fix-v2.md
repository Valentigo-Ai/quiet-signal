BUG HISTORY — READ THIS BEFORE TOUCHING THE KEYBOARD CODE AGAIN
1. **Fix attempt 1** (commit `9e7ad5ef`, shipped as version 10): `KeyboardAvoidingView` + a `setTimeout`/`scrollToEnd` on focus. Richard tested on his phone via the Internal Testing link and reported it was "still broken" on both screens.
2. **Fix attempt 2** (commit `3191790a`, shipped as version 11): rewrote to manually track keyboard height via `Keyboard.addListener('keyboardDidShow'/'keyboardDidHide')`, applying `endCoordinates.height` as dynamic bottom padding. This was verified by reading React Native's own Android source (`ReactRootView.java`) before shipping, on the theory that the JS-level height signal was accurate even under edge-to-edge. It still failed. Richard's follow-up screenshots showed the input box's on-screen position was **pixel-identical** whether the keyboard was open or closed — meaning the `keyboardDidShow` event was very likely never firing on his device at all, not just firing with a wrong height.
3. **Root cause, found this session by researching (not guessing) Android 15+ edge-to-edge behavior**: this app targets Android SDK 36, where edge-to-edge display is mandatory. Under edge-to-edge, `android:windowSoftInputMode="adjustResize"` (the default, and the thing both `KeyboardAvoidingView` and RN's own `Keyboard.addListener` legacy path depend on) **behaves like `adjustNothing`** — the OS no longer resizes the app window when the keyboard opens; it just hands over IME insets and expects the app to react to them directly. That's a documented, known-community-wide breaking change for exactly this kind of manual keyboard handling (see sources below). No amount of JS-side cleverness on top of `Keyboard.addListener` was ever going to fix this — the event itself isn't a reliable signal under edge-to-edge on all Android versions.
4. **Fix attempt 3** (commit `4466e353`, this session, NOT YET BUILT/TESTED): dropped all manual keyboard tracking in favor of `react-native-keyboard-controller` — the library Expo's own SDK docs now list and recommend specifically for this Android edge-to-edge keyboard problem. It reads the keyboard's position from a native `WindowInsetsAnimationCallback`, independent of `windowSoftInputMode`/window-resize behavior entirely, so it isn't affected by the edge-to-edge change that broke attempts 1 and 2.
